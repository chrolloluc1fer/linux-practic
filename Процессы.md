# 1. Что такое процесс

Процесс — это запущенная программа в памяти.

Когда ты выполняешь команду:

text

```
ls
```

происходит:

1. Bash вызывает `fork()` — создаётся копия текущего процесса.
2. В дочернем процессе вызывается `exec()` — загружается бинарник `ls`.
3. Процесс выполняется.
4. Родитель (bash) ждёт завершения через `wait()`.
5. Процесс завершается → освобождается память.

---

# 2. PID и PPID

- PID — идентификатор процесса.
- PPID — родительский PID.

Посмотреть:

text

```
ps aux
ps -ef
pstree -p
	- `ps -ef` — выведет плоский список всех запущенных процессов фулл формат с их PID и PPID.
- `pstree` — покажет то самое **дерево процессов**, наглядно демонстрируя, кто кого запустил.
- `top` или `htop` — отобразит процессы в реальном времени, показывая, сколько памяти и процессора они потребляют.
```

Все процессы в Linux образуют дерево.

PID 1 — корень дерева (обычно systemd).

---

# 3. Жизненный цикл процесса

Системные вызовы:

- `fork()` — создать копию процесса
- `exec()` — заменить код программы
- `wait()` — дождаться завершения

---

# 4. Состояния процессов

В `ps` можно увидеть:

| Состояние | Значение                                                                            |
| --------- | ----------------------------------------------------------------------------------- |
| R         | Running                                                                             |
| S         | Sleeping                                                                            |
| D         | Uninterruptible sleep (ожидание I/O) uninterruptible sleep (обычно ожидание диска). |
| Z         | Zombie                                                                              |
| T         | Stopped                                                                             |
| I         | Idle (бездействующий поток ядра)                                                    |

---

## Zombie (Z)

Процесс завершился,  
но родитель ещё не забрал статус через `wait()`.

У него:

- нет памяти
- нет кода
- осталась запись в таблице процессов

Если родитель умер → зомби “усыновляет” PID 1.

---

## Orphan (сирота)

Процесс, чей родитель завершился раньше него.

Становится дочерним процессом PID 1.

# 5. ps

text

```
ps aux
```

Колонки важные:

- USER
- PID
- %CPU
- %MEM
- STAT (состояние)
- COMMAND

Читаем колонки `ps aux`

👤 USER

**Кто запустил процесс.**

- Это имя пользователя в системе (например, `root`, `www-data`, `postgres` или ваше имя).
- **DevOps-совет:** Всегда обращайте внимание на эту колонку с точки зрения безопасности. Если веб-сайт запущен от пользователя `root` — это огромная дыра в безопасности. Если системный демон запущен от имени случайного пользователя — это повод для тревоги.

🆔 PID

**Уникальный идентификатор процесса (Process ID).**

- Тот самый номер паспорта процесса, о котором мы говорили в самом начале.
- **Зачем нужен:** Зная PID, вы можете управлять процессом. Например, если процесс завис, вы берёте этот номер и «убиваете» его командой `kill <PID>`.

⚡ %CPU

**Процент мощности процессора, который сейчас тратит процесс.**

- Показывает текущую нагрузку на процессор в данный момент времени.
- **Важный нюанс:** Если у вас на сервере 4 ядра, а процесс работает на полную мощность в одном потоке, он займет 100%. Если он нагрузит все 4 ядра по максимуму, вы увидите **400%**.

🧠 %MEM

**Процент оперативной памяти (RAM), занятой процессом.**

- Показывает долю от общего объема физической памяти сервера, которую отъел этот конкретный процесс.
- Если у вас 16 ГБ памяти, а процесс Java-приложения показывает 50%, значит, он один занял 8 ГБ.

🚦 STAT

**Текущее состояние процесса (Status).**  
Это самая интересная колонка для диагностики. Здесь вы увидите буквы, которые кодируют поведение процесса. Вот главные из них:

- **`R` (Running / Runnable):** Процесс прямо сейчас выполняется процессором или стоит в активной очереди и готов выполниться в любую микросекунду.
- **`S` (Interruptible Sleep):** Процесс спит. Он ждет какого-то события (например, когда пользователь нажмет клавишу или когда ядро пришлет сигнал `SIGCHLD`). Помнишь нашего родителя, который вызвал `wait()`? Вот он будет в состоянии `S`.
- **`D` (Uninterruptible Sleep):** Процесс в глубоком сне, его нельзя прервать. Обычно он ждет ответа от жесткого диска или сети (I/O). Если диск «умер» или сеть зависла, процесс намертво застревает в состоянии `D`, и его **невозможно убить даже через `kill -9`**, пока железка не ответит.
- **`Z` (Zombie):** Наш старый знакомый — зомби! Процесс уже мертв, но строка в таблице осталась, потому что родитель не вызвал `wait()`.
- **`T` (Stopped):** Процесс остановлен (поставлен на паузу). Например, если вы нажали комбинацию клавиш `Ctrl + Z` в терминале.

_Иногда рядом с буквой стоят знаки, например, `S+` (процесс работает на переднем плане терминала) или `Ss` (процесс является главным в сессии/сеансе)._

🛠️ COMMAND

**Какая именно команда или программа была запущена.**

- Здесь пишется путь к исполняемому файлу и все аргументы, с которыми его запустили.
- Например: `/usr/bin/python3 main.py` или `nginx: master process`.
- Если команда указана в квадратных скобках (например, `[ksoftirqd/0]`), это означает, что перед вами **поток ядра Linux**, а не обычная пользовательская программа.

---

Как это выглядит на практике (Пример)

text

```
USER       PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root         1  0.0  0.1 168012  9820 ?        Ss   10:00   0:02 /sbin/init
postgres  1234  0.5  4.2 450123 85000 ?        S    10:05   0:15 postgres: checkpointer
alex      5678 98.5  0.2  12456  4100 pts/0    R    11:20   5:40 python3 heavy_script.py
nginx     7777  0.0  0.0      0     0 ?        Z    11:30   0:00 [nginx] <defunct>
```

Используйте код с осторожностью.

**Что мы видим из этого лога как DevOps-инженеры:**

1. **PID 1** запущен от `root` — это система инициализации системы (`/sbin/init`), она мирно спит (`Ss`).
2. База данных **Postgres (PID 1234)** запущена от правильного пользователя `postgres`, потребляет адекватные 4.2% памяти и ожидает запросов (`S`).
3. Пользователь **alex (PID 5678)** запустил какой-то скрипт `heavy_script.py`. Скрипт прямо сейчас выполняется (`R`) и **сожрал почти целое ядро процессора (98.5% CPU)**. Это кандидат на проверку!
4. Процесс **Nginx (PID 7777)** превратился в зомби. На это указывает статус **`Z`** и пометка **`<defunct>`** (что переводится как «умерший/недействующий») в колонке COMMAND. Нужно искать его родительский процесс и разбираться, почему он не убирает за собой детей.

# 6. top

Показывает процессы в реальном времени.

Важно понимать:

- load average 
Эта метрика находится в самом верхнем правом углу. Выглядит она как три числа, например: `load average: 0.15, 0.85, 2.50`.

- **Что это такое:** Это среднее количество процессов, которые прямо сейчас выполняются процессором (`R`) или ждут своей очереди, а также процессы, застрявшие в ожидании диска (`D`).
- **Что значат три числа:** Это средняя нагрузка за последние **1 минуту**, **5 минут** и **15 минут** соответственно.
- **Как это читать (Правило моста):** Представь однополосный мост.
    - Если Load Average равен **1.0**, мост заполнен ровно на 100%. Машины едут, но свободного места нет.
    - Если равен **0.5**, мост полупустой.
    - Если равен **2.50**, на мосту пробка — одна машина едет, а еще полторы стоят в очереди.
- **Связь с ядрами:** Если у тебя **6 потоков/ядер** CPU (как мы выяснили в прошлом шаге), то лимит твоего "моста" — **6.0**.
- us / sy / id / wa
- **`us` (User):** Процент времени, который процессор тратит на пользовательские программы (твой код на Python, веб-сервер Nginx, база данных Postgres). Чем он выше при полезной работе — тем лучше.
- **`sy` (System):** Время, затраченное на работу самого ядра Linux (системные вызовы, выделение памяти, переключение контекста). Если `sy` аномально высокий (например, больше 30%), значит, программы слишком часто дергают ядро или в системе происходит какая-то внутренняя проблема.
- **`id` (Idle):** Процент полного бездействия. Процессор ничем не занят и просто отдыхает. Если `id` равен 95%, сервер скучает. Если `id` равен 0%, процессор загружен на максимум.
- **`wa` (iO Wait):** Время, которое процессор тратит на **ожидание ответа от жесткого диска или сети**. Это важнейший показатель для DevOps! Если `wa` высокий (больше 5–10%), это значит, что твой процессор мощный и готов работать, но он вынужден простаивать, потому что медленный диск (SSD/HDD) не успевает записывать или читать данные.


3. VIRT (Виртуальная память)

- **Суть:** Общий объём памяти, который процесс **запросил** у операционной системы «с запасом».
- **Нюанс:** Эта память еще не занята физически в железке. Цифра может быть огромной, пугаться её не стоит.

4. RES (Резидентная память)

- **Суть:** Объём **реальной физической оперативной памяти (RAM)**, которую процесс использует прямо сейчас.
- **Зачем смотреть:** Главный показатель для DevOps. Показывает истинный аппетит программы к оперативке.

# 8. Сигналы

Каждый процесс можно послать сигнал.

## Основные:

|Сигнал|Номер|Назначение|
|---|---|---|
|SIGTERM|15|нормальное завершение|
|SIGKILL|9|принудительное убийство|
|SIGINT|2|Ctrl+C|
|SIGHUP|1|перезапуск|

📋 Таблица главных сигналов для конспекта

|Сигнал|Номер|Как вызвать в консоли|Что делает / Особенности|
|---|---|---|---|
|**`SIGHUP`**|**1**|`kill -1 <PID>`|**Перезапуск (Hangup).** Говорит процессу перечитать свои конфигурационные файлы без полной остановки.|
|**`SIGINT`**|**2**|`Ctrl + C`|**Прерывание (Interrupt).** Вежливо просит остановить текущую операцию в терминале.|
|**`SIGTERM`**|**15**|`kill <PID>` _(по умолчанию)_|**Корректное завершение (Terminate).** Стандартный способ вежливо закрыть программу, дав ей время убрать за собой.|
|**`SIGKILL`**|**9**|`kill -9 <PID>`|**Принудительное убийство (Kill).** Мгновенное уничтожение процесса ядром. Программа не успевает ничего сделать.|

---

🧠 Подробный разбор: Как это работает изнутри

1️⃣ SIGHUP (Signal Hangup) — Номер 1

- **Исторический контекст:** Раньше этот сигнал отправлялся процессу, когда обрывалась связь с телефонным модемом (удаленный терминал «вешал трубку» — hang up).
- **Современное использование в DevOps:** Сейчас почти все серверные демоны (Nginx, Apache, SSHD) настроены так: когда они получают `SIGHUP`, они **не умирают**, а мгновенно на лету перечитывают свои файлы конфигурации.
- **Плюс для продакшна:** Это позволяет обновить настройки веб-сервера без перезагрузки самого сервера и без прерывания текущих сессий пользователей.

2️⃣ SIGINT (Signal Interrupt) — Номер 2

- **Как мы его используем:** Каждый раз, когда ты запускаешь в терминале бесконечную команду (например, `ping google.com` или тяжелый скрипт) и нажимаешь **`Ctrl + C`**, ты отправляешь процессу именно сигнал `SIGINT`.
- **Логика:** Это сигнал вежливого прерывания. Программа перехватывает его, аккуратно останавливает цикл и возвращает тебя в командную строку. Программа _может_ проигнорировать этот сигнал, если разработчик так её запрограммировал.

3️⃣ SIGTERM (Signal Terminate) — Номер 15

- **Главное правило:** Это **стандартный сигнал**, который отправляет команда `kill <PID>`, если не указывать никаких цифр.
- **Как это работает:** Это «джентльменская» просьба закрыться. Процесс получает сигнал, останавливает прием новых запросов, завершает текущие задачи, закрывает соединения с базой данных, сохраняет файлы на диск, удаляет свои временные файлы и только потом вызывает `exit()`.
- **Важно для DevOps:** При остановке контейнеров в Docker или Kubernetes система сначала всегда отправляет `SIGTERM`.

4️⃣ SIGKILL (Signal Kill) — Номер 9

- **Главное отличие:** Этот сигнал **нельзя перехватить, заблокировать или проигнорировать**.
- **Как это работает:** Если `SIGTERM` идет самому процессу, то `SIGKILL` идет **напрямую ядру Linux**. Ядро просто вышвыривает процесс из оперативной памяти и забирает его PID.
- **Опасность:** Процесс уничтожается мгновенно. Если в этот момент база данных записывала файл на диск, файл будет поврежден (Data Corruption). Временные файлы останутся на диске мусором.
- **Золотое правило сисадмина:** Никогда не начинай с `kill -9`. Сначала отправь вежливый `kill` (`SIGTERM`), подожди 5–10 секунд, и только если процесс намертво завис и не реагирует, применяй ядерное оружие — `kill -9`.

🔥 Пример из жизни (Аналогия с офисом)

Представь, что процесс — это бухгалтер, работающий за столом с документами.

- **`SIGHUP` (1):** Ты заходишь и говоришь: «Слушай, правила поменялись, вот новая инструкция от руководства, ознакомься». Бухгалтер читает её и продолжает работать.
- **`SIGINT` (2):** Ты говоришь: «Бросай считать этот отчет, он больше не нужен». Бухгалтер закрывает текущую папку.
- **`SIGTERM` (15):** Директор говорит: «Ваш рабочий день окончен, закрываем офис». Бухгалтер спокойно сохраняет Excel-файл, закрывает сейф, выключает компьютер, берет куртку и уходит.
- **`SIGKILL` (9):** В комнату врывается ОМОН, хватает бухгалтера под руки и уносит из здания. Компьютер остался включенным, сейф открыт, документы разбросаны по полу.

# 9. nice / renice

Позволяет менять приоритет процесса.

Когда процессов в системе много, планировщик Linux должен решать, кому из них дать процессорное время в первую очередь. Для этого у каждого процесса есть показатель «вежливости» — **Nice value**.

- **Диапазон:** от **`-20`** до **`19`**.
- **Дефолтное значение:** Новый процесс по умолчанию запускается с приоритетом **`0`**.

⚖️ Главное правило: Больше число ➡️ Ниже приоритет

Это кажется контринтуитивным, но логика здесь заложена в самом слове _nice_ (вежливый):

- **Чем выше число (например, `19`), тем процесс «вежливее».** Он говорит другим процессам: «Пожалуйста, проходите вперёд, мне не к спеху». Такой процесс будет получать процессорное время в последнюю очередь.
- **Чем ниже число (например, `-20`), тем процесс «наглее» и агрессивнее.** Он расталкивает остальные процессы локтями и требует запустить его немедленно.

> 🔒 **Ограничение безопасности:** Любой обычный пользователь может сделать свой процесс более «вежливым» (повысить nice до 19). Но делать процессы «наглыми» (опускать nice ниже 0) может **только суперпользователь (`root`)**, чтобы обычные юзеры не могли монополизировать процессор.

---

Диапазон nice:

text

```
-20 (высший приоритет)
до
19 (низший)
```

Запуск:

text

```
nice -n 10 command
```

Изменить:

text

```
renice 5 -p PID
```


📊 Как увидеть Nice value в системе?

- В выводе команды `top` приоритет процесса пишется в колонке **`NI`**.
- В выводе команды `ps` его можно увидеть, добавив флаги, например, `ps -eo pid,ni,comm`.

# 10. /proc

Псевдофайловая система с информацией о процессах.

Пример:

text

```
ls /proc
cat /proc/1234/status
cat /proc/1234/cmdline
```

📁 Что такое `/proc`?

- **Псевдофайловая система:** Она виртуальна, не существует на жестком диске и не занимает место.
- **Окно в ядро:** Существует только в оперативной памяти. Ядро Linux генерирует текст в файлах «на лету» в момент их открытия.
- **Источник данных:** Утилиты `ps`, `top` и `htop` собирают всю информацию о системе именно отсюда.

🗂️ Структура каталога

- По команде `ls /proc` выводится список папок.
- Папки с **цифровыми именами** (например, `/proc/1234/`) содержат данные о процессах, где имя папки — это **PID** процесса.

🛠️ Главные команды для работы (на примере PID 1234)

- **`ls /proc`** — посмотреть список всех процессов (их PID) и системных параметров.
- **`cat /proc/1234/status`** — вывести подробный статус процесса (Имя, состояние `State`, родительский `PPID`, реальное потребление памяти `VmRSS`).
- **`cat /proc/1234/cmdline`** — показать точную команду, которой был запущен процесс.
    - _Лайфхак:_ Так как аргументы внутри разделены нулевым байтом (`\0`), для красивого вывода с пробелами используй:  
        `cat /proc/1234/cmdline | tr '\0' ' '`

💡 Дополнительные полезные файлы процесса

- `/proc/1234/exe` — ссылка на реальный исполняемый файл на диске.
- `/proc/1234/fd/` — папка со всеми открытыми процессом файлами и сетевыми сокетами (дескрипторами).

# 11. OOM Killer

Когда заканчивается память,  
ядро выбирает процесс для убийства.

Параметр:

text

```
oom_score
oom_score_adj
```

**OOM Killer** — это защитный механизм ядра Linux «последней надежды». Он активируется, когда в системе заканчивается физическая оперативная память (RAM) и раздел подкачки (Swap), спасая сервер от полного зависания (Kernel Panic).

🛠️ Как он работает

1. Когда свободная память равна 0% и выделить её больше невозможно, просыпается OOM Killer.
2. Он выбирает один процесс по специальному алгоритму и мгновенно уничтожает его самым жестким сигналом **`SIGKILL` (9)**.
3. Память убитого процесса возвращается системе, и сервер продолжает жить.

🧮 Балл смертника: `oom_score`

Ядро выбирает жертву на основе очков «виновности» процесса — **`oom_score`** (лежит в `/proc/PID/oom_score`).

- **Чем выше число, тем выше шанс, что процесс убьют.**
- **Главный фактор:** Объём занимаемой памяти (у кого больше `RES` в `top`, у того выше балл).
- **Смягчающие факторы:** Процессам от `root` и старым (долгоживущим) процессам балл немного снижается.

🛡️ Защита важных процессов: `oom_score_adj`

Файл `/proc/PID/oom_score_adj` позволяет DevOps-инженеру вручную скорректировать балл процесса в диапазоне от **`-1000`** до **`1000`**.

- Запись значения **`-1000`** даёт процессу **полный иммунитет** — OOM Killer никогда его не тронет. Применяется для критически важных баз данных (PostgreSQL, MySQL).

bash

```
# Полностью защитить базу данных с PID 1234 от OOM Killer:
echo -1000 > /proc/1234/oom_score_adj
```

Используйте код с осторожностью.

🩺 Как расследовать убийство процессом OOM

Если процесс (например, Java или Python) внезапно исчез, ищи следы «казни» в системных логах:

bash

```
dmesg -T | grep -i oom
# или
grep -i kill /var/log/syslog
```

Используйте код с осторожностью.

_Если OOM Killer работал, ты увидишь системную строчку с текстом: `Out of memory: Kill process...`_

# 12. systemd как PID 1

PID 1:

- запускает сервисы
- усыновляет сирот
- управляет системой

Проверить:

text

```
ps -p 1
```

- **`systemd` (PID 1)** — первый процесс в системе, запускаемый ядром. Главный корень дерева процессов.
- **Смерть PID 1** = мгновенный крах системы (Kernel Panic).
- **Главные обязанности:**
    1. Запуск и координация всех системных служб при старте.
    2. Автоматическое усыновление процессов-сирот и сбор их статусов (защита от зомби).
    3. Мониторинг и автоматический перезапуск упавших сервисов (по правилам из unit-файлов).
- **Команда проверки:** `ps -p 1`

14️⃣ Поиск процесса по порту

Когда нужно узнать, какая именно программа заняла сетевой порт (например, чтобы освободить его или проверить, запустился ли твой веб-сервер), используются специальные сетевые утилиты.

> 🔒 **Важное правило:** Все команды поиска по портам **обязательно запускать через `sudo`**. Без прав суперпользователя Linux из соображений безопасности скроет имена процессов и их PID.

🛠️ Способ 1: Современный стандарт (`ss`)

Утилита `ss` (Socket Statistics) — самая быстрая и современная замена старой команде `netstat`.

bash

```
sudo ss -tulpn | grep :80
```

Используйте код с осторожностью.

_(Вместо `80` подставь нужный порт, например `:8080`, `:5432`)_

**Разбор флагов:**

- `-t` / `-u` — искать только TCP / UDP соединения.
- `-l` (listening) — показать только те порты, которые прямо сейчас слушают входящие запросы.
- `-p` (processes) — **вывести PID и имя процесса**, занявшего порт.
- `-n` (numeric) — показывать порты цифрами (80), а не именами протоколов (http).

🛠️ Способ 2: Чистый вывод через файлы (`lsof`)

Утилита `lsof` (List Open Files) ищет сокет как открытый виртуальный файл. Она выдает очень чистый результат без лишней системной информации.

bash

```
sudo lsof -i :80
```

Используйте код с осторожностью.

**Разбор флага:**

- `-i :80` — жестко фильтрует вывод по конкретному сетевому порту. Сразу показывает колонки `COMMAND` и `PID`.

📋 Пример анализа вывода

Допустим, после ввода `sudo ss -tulpn | grep :8080` ты получил строку:

text

```
tcp   LISTEN 0      4096         0.0.0.0:8080       0.0.0.0:*    users:(("java",pid=5678,fd=14))
```

Используйте код с осторожностью.

**Твои действия как инженера:**

1. Ты видишь, что порт `8080` занят процессом **`java`** с **PID 5678**.
2. Теперь ты можешь изучить его: `cat /proc/5678/status`
3. Или вежливо завершить его, чтобы освободить порт: `kill 5678`
