1. **UID (User ID)**
    
    - У каждого пользователя в Linux есть числовой идентификатор (UID). Обычно для системных пользователей UID маленький (до 1000), а для обычных — начинается с 1000 или 500 (зависит от дистрибутива).
    - Зная UID, система понимает, под каким именем запущен процесс или обрабатывается файл.
2. **GID (Group ID)**
    
    - Похожая логика действует и для групп. У каждой группы есть числовой идентификатор (GID).
    - Пользователь может состоять в нескольких группах — основной (primary group) и дополнительных (secondary groups).
3. **/etc/passwd и /etc/shadow**
    
    - Файл `/etc/passwd` хранит основные данные о пользователях (логин, UID, GID, домашняя директория, используемый shell).
    - Файл `/etc/shadow` содержит зашифрованные пароли и информацию о политике паролей (срок действия, дата последней смены и т.д.).

```
sudo -l - какие команды разрешены

sudo useradd -r -s /usr/sbin/nologin nginx
```

**Как работают эти флаги:**

- **`-r` (system)** — указывает утилите создать именно _системного_ пользователя. Ему автоматически присвоится свободный UID меньше 1000.
- **`-s /usr/sbin/nologin` (shell)** — устанавливает «заглушку» вместо стандартной командной оболочки. Если кто-то попытается войти на сервер под аккаунтом `nginx`, система мгновенно оборвёт сессию.

Права `rwx` для директорий работают немного иначе, чем для файлов:

1. **r (read)**
    
    - Разрешает просматривать список содержимого директории (имена файлов).
    - Без `x` (execute) вы видите список имён, но не сможете «войти» в каталог (cd).
2. **w (write)**
    
    - Позволяет создавать, удалять и переименовывать файлы внутри директории.
    - Если у вас есть `w`, но нет `r`, вы можете добавлять/удалять файлы, не видя полного списка.
3. **x (execute)**
    
    - Разрешает «вход» в директорию (команда `cd`).
    - Без `r` и `x` вы не сможете даже проверить, что внутри, а без `x` не сможете перейти в каталог, даже если вы видите его содержимое.

### 2.1 `groupadd`

**Назначение**  
Создаёт новую группу в системе. Обычно указывается имя группы; при необходимости можно задать определённый GID (Group ID).

**Синтаксис**

```css
groupadd [опции] <имя_группы>
```

- **Пример**:
    
    ```nginx
    groupadd devteam
    ```
    
    Создаёт группу `devteam` с автоматически сгенерированным GID.

### 2.2 `groupdel`

**Назначение**  
Удаляет группу из системы. Применяется, когда группа больше не используется или когда нужно избежать накопления «мертвых» записей.

**Синтаксис**

```php-template
groupdel <имя_группы>
```

- **Важно**: перед удалением стоит проверить, нет ли пользователей, чьей основной группой является данная. Если она основная, необходимо сначала сменить основную группу пользователей (через `usermod`).

### 2.3 `gpasswd`

**Назначение**  
Управляет паролем группы и её членами. Когда группа защищена паролем, пользователи могут временно присоединяться к ней через команду `newgrp`, введя пароль. Это более старая модель, которая реже используется, но иногда встречается.

**Пример**

```nginx
gpasswd devteam
```

Попросит установить пароль для группы `devteam`.

Кроме того, `gpasswd -a <user> <group>` добавляет пользователя в группу, а `gpasswd -d <user> <group>` удаляет пользователя из группы.

### 2.4 `usermod -aG`

**Назначение**  
Позволяет добавить пользователя в дополнительную (вторичную) группу, не затрагивая его основную группу и прочие настройки.

**Синтаксис**

```php-template
usermod -aG <имя_группы> <имя_пользователя>
```

- **Опция `-a`** (append) означает «добавить пользователя в указанные группы, не убирая его из других». Без `-a` пользователь будет исключён из всех групп, кроме перечисленных.
- **Пример**:
    
    ```nginx
    usermod -aG devteam alice
    ```
    
    Пользователь `alice` получит дополнительный доступ, который есть у группы `devteam`.

Утилита `chage` (от _change age_) регулирует срок действия пароля, дату последней смены и уведомления пользователя о необходимости сменить пароль. Она взаимодействует с информацией в `/etc/shadow`.

### Синтаксис

```css
chage [опции] <имя_пользователя>
```

- **Опции**:
    - `-l` (list) — показать текущие настройки срока действия пароля.
    - `-d <день>` — задать дату последней смены пароля.
    - `-E <день>` — установить дату, когда учётная запись будет заблокирована (по формату YYYY-MM-DD).
    - `-M <число>` — задать максимальное число дней, через которое пароль устареет.
    - `-W <число>` — за сколько дней до истечения пароля предупреждать пользователя.

### Пример

```nginx
chage -M 90 -W 7 alice
```

- Устанавливает максимальное время действия пароля для пользователя `alice` в 90 дней и начинает предупреждать за 7 дней до истечения.

Файл `/etc/login.defs` задаёт глобальные параметры, которые влияют на создание и обслуживание пользователей. В нём можно настроить:

- **Диапазон UID и GID** для новых аккаунтов;
- **Условия хранения паролей** (минимальное количество символов, срок действия и т.д.);
- **Параметры блокировки** учётной записи;
- **Директивы** о том, каким образом система обрабатывает логины по умолчанию.

- **Файлы конфигурации**:
    
    - `~/.bashrc`, `~/.bash_profile` (или `~/.profile`) для Bash;
    - `~/.zshrc` для Zsh.
    
    В них задают переменные среды (`PATH`, `PS1`), алиасы, функции оболочки и прочие настройки.
- **Глобальные настройки** расположены в `/etc/bashrc`, `/etc/profile` и т. д. — применяются ко всем пользователям.

### 3.2 Базовые настройки домашней директории

Когда вы создаёте нового пользователя, система обычно копирует шаблонные файлы из `/etc/skel` в домашнюю директорию (например, `~/.bashrc`, `~/.profile`). Таким образом у пользователя появляется базовый набор конфигурационных файлов.


## 2. Как установить и проверить текущий umask

### Проверка

Чтобы узнать, какой umask активен в текущей сессии, достаточно выполнить команду:

```bash
umask
```

- Обычно покажет результат в виде четырёхзначного (иногда трёхзначного) числа, например `0022` или `002`.

### Установка

- Команда `umask <значение>` меняет umask для текущей shell-сессии.
    
    ```bash
    umask 002
    ```
    
    Теперь все новые файлы и папки будут создаваться с учётом этой маски, **но** только в рамках текущего сеанса.
- Если вы хотите, чтобы umask устанавливался при каждом входе пользователя, нужно прописать команду `umask <значение>` в его файлах инициализации shell (например, `~/.bashrc`, `~/.profile`) или в глобальных скриптах (`/etc/profile`, `/etc/bashrc` и т. д.).

### 1.1 Что такое SUID

**SUID (Set User ID)** — это специальный бит, который можно установить на исполняемый файл. Когда пользователь запускает такой файл:

- Программа работает с **эффективным UID** владельца файла, а не UID того, кто запустил программу.
- Проще говоря, если файл принадлежит root и имеет установленный бит SUID, то **любое** лицо, запустившее эту программу, временно получает права владельца (root) **в рамках** процесса.

**Пример**

- `/usr/bin/passwd`: принадлежит root и имеет бит SUID.
    - Когда обычный пользователь запускает `passwd`, программа меняет пароль, обращаясь к системным файлам вроде `/etc/shadow`, которые обычно доступны только root.
    - Это работает, потому что процесс `passwd` во время выполнения действует «от лица root», хотя запущен обычным пользователем.

#### Как узнать, стоит ли SUID

- В выводе `ls -l` вместо обычного `x` у владельца (user) вы увидите символ `s`. Например, `-rwsr-xr-x`.
- Ставится SUID так:
    
    ```bash
    chmod u+s файл
    ```
    
    или в октальной форме `chmod 4755 файл` (если у файла обычные права 755).

### 1.2 Что такое SGID

**SGID (Set Group ID)** — похож на SUID, но относится к группе:

- Если SGID установлен на исполняемом **файле**, то при запуске программа получает **эффективный GID** той группы, которая владеет файлом.
- Если SGID установлен на **директории**, то все новые файлы, созданные внутри этой директории, унаследуют **группу** этой директории, а не основную группу пользователя, который создал файл.

**Пример**

- Если у нас есть папка `shared/` с правами `rwxrwsr-x` (то есть SGID-бит у группы), то любые файлы, созданные внутри `shared/`, будут принадлежать группе `sharedgroup` (если именно она владеет этой директорией).
- Это удобно в командной работе, когда все члены проекта хотят автоматически иметь одну группу у новых файлов, чтобы можно было совместно редактировать их.

#### Как узнать, стоит ли SGID

- В выводе `ls -l` на месте бита x для группы (g) вы увидите символ `s`. Например, `drwxr-sr-x`.
- Команда установки SGID для каталога:
    
    ```bash
    chmod g+s каталог
    ```
    
    В октальной форме: `chmod 2775 каталог` (если обычные права — 775).


- **SUID** = **4** [1] s
- **SGID** = **2** [1] s
- **Sticky bit** = **1** [1] t
- **Маленькая `s`** (`rws`) — спецфлаг активен, и у пользователя/группы **есть** право на выполнение (`x`). Всё работает штатно.
- **Большая `S`** (`rwS`) — спецфлаг активен, но у пользователя/группы **нет** права на выполнение (`x`). В таком случае SUID или SGID **работать не будут**, так как файл в принципе нельзя запустить.

## 2. Sticky bit для директорий

### 2.1 Что это такое

**Sticky bit** изначально применялся к файлам, чтобы оставлять часть программы в памяти для ускорения запуска, но в современном Linux его чаще используют **на директориях**. Sticky bit в директории означает:

- Только **владелец** файла (либо директории), либо root может удалить или переименовать файл внутри этой директории, даже если у директории есть права на запись всем (`rwxrwxrwt`).

**Пример: `/tmp`**

- У директории `/tmp` вы увидите права `drwxrwxrwt`:
    - `rwx` — для владельца
    - `rwx` — для группы
    - `rwxt` — для остальных (обратите внимание на `t` вместо обычного `x`)
- Это значит, что `/tmp` доступна всем на запись (любой может создать файл), **но** пользователь может удалить **только** свой собственный файл. Это защищает от ситуации, когда один пользователь случайно или преднамеренно трёт файлы другого.

#### Как узнать, стоит ли Sticky bit

- В выводе `ls -l` вместо x у «others» появляется буква `t`. Например: `drwxrwxrwt`.
- Установка Sticky bit:
    
    ```bash
    chmod +t каталог
    ```
    
    или в октальной форме `chmod 1777 каталог` (если обычные права 777).

### 2.1 Просмотр ACL: `getfacl`

**getfacl** показывает расширенные права, которые применяются к файлу или директории.

```php-template
getfacl <файл_или_каталог>
```

**Пример вывода**:

```makefile
# file: report.txt
# owner: user1
# group: devgroup
user::rw-
user:user2:rw-
group::r--
mask::rw-
other::r--
```

- **user::rw-** — базовые права владельца (user1).
- **user:user2:rw-** — конкретной учётной записи user2 дали чтение и запись (rw-).
- **group::r--** — базовые права группы devgroup.
- **mask::rw-** — максимальный набор прав, который может быть у дополнительных записей ACL (об этом ниже).
- **other::r--** — права для всех прочих пользователей.

### 2.2 Изменение ACL: `setfacl`

**setfacl** позволяет добавлять, менять или удалять записи ACL.

#### Основной синтаксис

```php-template
setfacl -m <ACE> <файл_или_каталог>
```

- `-m` (modify) — модифицирует текущий список ACL.
- `ACE` (Access Control Entry) — запись вида `u:<user>:<rwx>` или `g:<group>:<rwx>`.

**Пример**

```applescript
setfacl -m u:user2:rw file.txt
```

- Это добавляет (или меняет) запись в ACL: **пользователь user2** будет иметь `r` и `w` на `file.txt`.

Если нужно убрать конкретную запись, используют `-x`:

```css
setfacl -x u:user2 file.txt
```

- Это удаляет ACL-запись для user2.

#### Рекурсивное применение

Чтобы рекурсивно задать ACL для каталога (и всего содержимого), добавляют флаг `-R`:

```bash
setfacl -R -m u:user2:rw /shared_dir/
```

### 3.1 Наследование в директориях

При работе с директориями и ACL можно настроить, чтобы вновь создаваемые файлы и поддиректории **унаследовали** те же расширенные права. Для этого используется специальный «default ACL».

- `setfacl -m d:u:user2:rw /shared_dir/`
- Буква `d` (default) означает «по умолчанию» для новых файлов в этой директории.

Таким образом, любые файлы или поддиректории, созданные внутри `/shared_dir/`, изначально будут иметь ACL-запись, которая разрешает user2 читать и писать.

### 3.2 Маска (mask)

В выводе `getfacl` вы могли заметить строку **mask::**. Маска ограничивает максимальный набор прав, которые могут применяться к «дополнительным» записям (ACL).

- Если в маске стоит `rw-`, то даже если конкретному пользователю назначено `rwx`, **execute** не будет действовать, потому что маска не разрешает `x`.
- Маска не влияет на базовые права влад
- ельца (user::) и на «others::», но влияет на group:: и на все дополнительные user: / group: ACL-записи.


- `-m mask:rwx file` — задать `mask` явно для файла
- `-d -m mask:rwx dir` — задать `mask` явно для директории


Файл **`/etc/sudoers`** и директивы в **`/etc/sudoers.d/`** определяют, **кто** (какие пользователи и группы) может выполнять **какие** команды с правами root (или других пользователей). Изменяется с помощью команды `visudo`, которая проверяет синтаксис, чтобы избежать ошибок.

**Пример фрагмента** в `/etc/sudoers`:

```sql
# Разрешаем пользователю alice запускать все команды от имени root, запрашивая её пароль
alice ALL=(ALL:ALL) ALL

# Разрешаем группе admin запускать все команды без пароля
%admin ALL=(ALL:ALL) NOPASSWD:ALL

# Разрешаем bob только перезагружать nginx
bob ALL=(root) /bin/systemctl restart nginx
```

- **ALL=(ALL:ALL)** означает, что пользователь (alice) может действовать как любой пользователь и любая группа.
- **NOPASSWD** — отключает запрос пароля для данного пользователя/группы.
- Конкретная команда `/bin/systemctl restart nginx` — точечное разрешение для bob.
1. **Системные журналы**
    
    - Большинство дистрибутивов ведут лог аутентификаций в `/var/log/auth.log` (Debian/Ubuntu) или `/var/log/secure` (RHEL/CentOS).
    - Здесь записываются события: успешный и неуспешный вход, запуск sudo, переключение пользователя через `su`.
    - Анализ логов помогает выяснить, кто и когда входил в систему, были ли странные попытки входа.
2. **Утилита `last`**
    
    - Показывает историю последних сеансов (logins) пользователей.
    - Можно быстро узнать, кто входил, откуда (IP-адрес), когда завершил сеанс.
    - Пример использования:
        
        ```css
        last -a
        ```
        
        (Покажет и IP-адрес/hostname, откуда логинились.)
3. **auditd** (Linux Auditing System)
    
    - Более продвинутая система аудита, которая может отслеживать **конкретные** события: попытки открыть важный файл, изменение конфигурационных директорий и т. п.
    - Конфигурируется через `/etc/audit/audit.rules` или `/etc/audit/rules.d/`.
    - Позволяет настраивать фильтры: «логировать все изменения в `/etc/passwd`, `/etc/shadow`», «все обращения к файлам в `/var/www`» и т. п.
    - Полезно для расследований, мониторинга действий с высокими привилегиями.
4. **Сбор и централизованный анализ логов**
    
    - В крупных компаниях логи с серверов отправляют на центральные системы (ELK Stack, Graylog, Splunk), где их можно фильтровать, сортировать, искать корреляции.
    - Если DevOps-инженер замечает, что кто-то под root делал подозрительные действия, можно быстро «пробить» это через систему логирования.
